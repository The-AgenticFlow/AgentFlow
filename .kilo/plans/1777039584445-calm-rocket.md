# Error Feedback Mechanism for Forge Pairs

## Problem

Errors during the forge pair lifecycle are swallowed or only logged, never fed back to agents so they can reason about and resolve them. This leads to:

1. **Worktree setup failures silently ignored** — `git fetch/merge origin/main` failures in `create_worktree()` (worktree.rs:44-49) are warned but the pair continues, spawning FORGE into a broken workspace
2. **Merge failures during rework are swallowed** — When `merge_origin_main()` fails during conflict/CI rework (pair.rs:447-453, 480-486), FORGE is spawned without knowing the merge failed
3. **No post-completion verification** — FORGE may write code that doesn't compile/build, and the only feedback loop is the slow VESSEL→CI round-trip (10+ minutes). When verification fails, the agent has no chance to self-correct before being routed away
4. **Error context lost between attempts** — When the pair retries or re-enters the flow, previous error context is discarded. The agent can't see what went wrong before and may repeat the same mistakes

The existing `CONFLICT_RESOLUTION.md` and `CI_FIX.md` patterns prove that context-file-driven error feedback works well. FORGE receives structured error context and can reason about it. We need a **generalized error feedback mechanism** that captures errors from any phase of the pair lifecycle, feeds them back to FORGE for self-repair, and escalates to nexus if unresolvable.

## Design Philosophy

**The agent should always see its errors.** When any error occurs during the pair lifecycle — setup, git operations, build, test, or any other failure — the error output must be:
1. **Captured** — Not just logged, but preserved in a form FORGE can read
2. **Fed back** — Written to a context file that FORGE receives in its prompt
3. **Actionable** — Include resolution hints so FORGE knows what to try
4. **Bounded** — After N failed self-repair attempts, escalate to the orchestrator (nexus)

This creates a **self-healing loop within the pair lifecycle** with escalation as the fallback:

```
Error occurs → capture error output → write ERROR_FEEDBACK.md → 
FORGE sees error context → FORGE attempts fix → 
  success → continue normal flow
  failure → increment attempt counter → 
    under max → respawn FORGE with error context + history
    at max → return PairOutcome::Blocked with detailed error → route to nexus
```

## Implementation

### 1. Generalized Error Feedback Context File (`ERROR_FEEDBACK.md`)

Instead of separate SETUP_ERROR.md / BUILD_ERROR.md files, use a single unified `ERROR_FEEDBACK.md` that captures errors from **any phase** of the pair lifecycle. This follows the same pattern as CI_FIX.md and CONFLICT_RESOLUTION.md, but is more general.

The pair lifecycle already checks for specific context files (CONFLICT_RESOLUTION.md, CI_FIX.md) and modifies behavior accordingly. ERROR_FEEDBACK.md works the same way but covers all other error cases.

**Format:**
```markdown
# Error Feedback — Self-Repair Required

An error occurred during the pair lifecycle that you must resolve.

## Error Source
setup / build_verification / git_operation / process_error

## Error Output
```
<raw error output — stderr, git error, build failure, etc.>
```

## Context
- Branch: forge-1/T-015
- Worktree: /path/to/worktree
- Phase: conflict_rework
- Attempt: 2 of 3

## Resolution Hints
- If git merge failed with unmerged files: run `git merge --abort`, then `git fetch origin main && git merge origin/main --no-edit`
- If build failed: read the error output, fix ALL errors, then run the build command again

## Previous Attempts
- Attempt 1: git merge --abort then re-merge — resolved unmerged files but new conflict appeared
(See error_history.json for full details)

## Instructions
1. Assess the error output above
2. Take the suggested resolution steps
3. If you cannot resolve after reasonable effort, write STATUS.json with status BLOCKED and include the reason
4. If you resolve the error, continue with your original task and write STATUS.json normally
```

#### 1g. ERROR_FEEDBACK.md lifecycle — write current error only, clear on resolution

**Critical design rule**: ERROR_FEEDBACK.md must always reflect **only the current, unresolved error**. When a new error occurs, it **overwrites** (not appends to) the previous file. When the error is resolved, the file is **removed immediately** — not at end-of-lifecycle.

This prevents stale error context from bleeding into subsequent FORGE prompts and avoids FORGE trying to fix an old error that's already been resolved, or conflating it with a new error.

**Clear on resolution** — called whenever FORGE produces a terminal outcome that passes verification, or when the pair transitions out of error state:

```rust
async fn clear_error_feedback(&self) -> Result<()> {
    let path = self.config.shared.join("ERROR_FEEDBACK.md");
    if path.exists() {
        tokio::fs::remove_file(&path).await?;
        debug!("Removed ERROR_FEEDBACK.md — error resolved");
    }
    Ok(())
}
```

**Integration points for clearing:**
1. `read_status()` — when FORGE writes a terminal status AND verification passes: clear ERROR_FEEDBACK.md before returning the outcome
2. `provision_worktree()` — if called on a resume and no warnings this time: clear any stale ERROR_FEEDBACK.md
3. `cleanup()` — always clear as a safety net (belt and suspenders):

```rust
// In cleanup() — safety net, should already be cleared on resolution
let error_feedback_path = self.config.shared.join("ERROR_FEEDBACK.md");
if error_feedback_path.exists() {
    let _ = tokio::fs::remove_file(&error_feedback_path).await;
    debug!("Removed ERROR_FEEDBACK.md in cleanup (safety net)");
}
```

**Write always overwrites** — `write_error_feedback()` uses `tokio::fs::write()` which replaces the entire file, never appends. Each invocation represents one current error, not a log of past errors. The error history is tracked separately in `error_history.json`.

### 2. Post-Completion Verification Hook

After FORGE writes STATUS.json claiming completion (PR_OPENED, COMPLETE), run a **configurable verification step** before accepting the outcome. If verification fails, write ERROR_FEEDBACK.md and respawn FORGE for self-repair.

**Key principle**: The verification command is NOT auto-detected from package.json conventions. It is **configured by the project**. This avoids the fragility of guessing build commands.

#### 2a. Verification configuration

Add to `PairConfig`:
```rust
pub struct PairConfig {
    // ... existing fields ...
    
    /// Command to run for post-completion verification.
    /// Configured via AGENTS.md or .agentflow config.
    /// If None, verification is skipped.
    pub verify_command: Option<String>,
    
    /// Maximum number of verification failure → self-repair cycles.
    pub max_verify_attempts: u32,
}
```

Configuration sources (in priority order):
1. `PairConfig::verify_command` — set programmatically
2. `AGENTS.md` in the project root — parse a `<!-- verify: npm run build && npm test -->` comment or a `## Verify Command` section
3. `.agentflow/verify.sh` — if this script exists in the worktree, run it
4. **None** — skip verification entirely (current behavior)

This is intentionally opt-in. Projects that don't configure verification continue working exactly as before. Projects that do get the self-healing loop.

#### 2b. `ForgeSentinelPair::verify_completion()` method

```rust
struct VerificationState {
    attempt: u32,
    max_attempts: u32,
}

impl ForgeSentinelPair {
    async fn verify_completion(&self) -> Result<VerificationResult> {
        let command = match &self.config.verify_command {
            Some(cmd) => cmd.clone(),
            None => return Ok(VerificationResult::Skipped),
        };
        
        info!(command = %command, "Running post-completion verification");
        
        let output = tokio::process::Command::new("sh")
            .arg("-c")
            .arg(&command)
            .current_dir(&self.config.worktree)
            .output()
            .await
            .context("Failed to run verification command")?;
        
        if output.status.success() {
            info!("Verification passed — accepting FORGE completion");
            Ok(VerificationResult::Passed)
        } else {
            let stderr = String::from_utf8_lossy(&output.stderr);
            let stdout = String::from_utf8_lossy(&output.stdout);
            let combined = format!("{}\n{}", stdout, stderr);
            warn!(command = %command, "Verification failed — feeding error back to FORGE");
            Ok(VerificationResult::Failed { output: combined, command })
        }
    }
}

enum VerificationResult {
    Passed,
    Failed { output: String, command: String },
    Skipped,
}
```

#### 2c. Integrate into `read_status()` flow

In `ForgeSentinelPair::read_status()`, after parsing a terminal outcome, run verification. **On success, clear ERROR_FEEDBACK.md immediately** so it doesn't persist into the next lifecycle:

```rust
// After determining outcome from STATUS.json...
if matches!(outcome, PairOutcome::PrOpened { .. } | PairOutcome::Blocked { .. }) {
    match self.verify_completion().await? {
        VerificationResult::Passed => {
            // Error resolved — clean up so next lifecycle doesn't see stale context
            self.clear_error_feedback().await?;
            self.mark_error_resolved().await?;
            return Ok(Some(outcome));
        }
        VerificationResult::Failed { output, command } => {
            self.verification_state.attempt += 1;
            
            if self.verification_state.attempt >= self.verification_state.max_attempts {
                warn!(
                    attempts = self.verification_state.attempt,
                    "Max verification attempts reached — escalating to nexus"
                );
                return Ok(Some(PairOutcome::Blocked {
                    reason: format!(
                        "Verification failed {} times. Last error from `{}):\n{}",
                        self.verification_state.attempt, command, output
                    ),
                    blockers: vec![Blocker {
                        blocker_type: "verification_failure".to_string(),
                        description: format!("Post-completion verification failed: {}", command),
                        nexus_action: "Review verification errors and decide whether to re-assign or close".to_string(),
                    }],
                }));
            }
            
            // Feed error back to FORGE for self-repair
            // write_error_feedback OVERWRITES — this is the current error only
            self.write_error_feedback(
                "build_verification",
                &format!("Verification command `{} failed:\n{}", command, output),
                Some(&format!(
                    "Run `{} locally, fix ALL errors, then commit and push",
                    command
                )),
            ).await?;
            
            // Append to error history
            self.append_error_history("build_verification", &output).await?;
            
            // Archive the non-terminal STATUS.json so the event loop continues
            self.archive_non_terminal_status(&normalized).await?;
            return Ok(None); // event loop will respawn FORGE with error context
        }
        VerificationResult::Skipped => return Ok(Some(outcome)),
    }
}
```

#### 2d. `write_error_feedback()` — unified error context writer (overwrites on each call)

**Key invariant**: Each call to `write_error_feedback` **overwrites** the file completely. It never appends. The file always contains exactly one error — the current one. This prevents stale error context from mixing with new errors or persisting after resolution.

```rust
/// Write ERROR_FEEDBACK.md with the current error context.
/// OVERWRITES any existing ERROR_FEEDBACK.md — this file represents
/// ONE current error, not a log of past errors.
/// Past errors are tracked in error_history.json.
async fn write_error_feedback(
    &self,
    source: &str,
    error_output: &str,
    hint: Option<&str>,
) -> Result<()> {
    let attempt = self.error_feedback_attempts + 1;
    let branch = WorktreeManager::branch_name(&self.config.pair_id, &self.ticket_id);
    
    let history_section = self.format_error_history_section().await?;
    
    let content = format!(
        "# Error Feedback — Self-Repair Required\n\n\
         An error occurred during the pair lifecycle that you must resolve.\n\n\
         ## Error Source\n{source}\n\n\
         ## Error Output\n```\n{error_output}\n```\n\n\
         ## Context\n\
         - Branch: {branch}\n\
         - Worktree: {worktree}\n\
         - Attempt: {attempt} of {max_attempts}\n\n\
         {hint_section}\
         {history_section}\
         ## Instructions\n\
         1. Assess the error output above\n\
         2. Take the suggested resolution steps (if any)\n\
         3. If you cannot resolve, write STATUS.json with status BLOCKED\n\
         4. If you resolve the error, continue with your task and write STATUS.json normally\n",
        source = source,
        error_output = error_output,
        branch = branch,
        worktree = self.config.worktree.display(),
        attempt = attempt,
        max_attempts = self.config.max_verify_attempts,
        hint_section = hint.map(|h| format!("## Resolution Hints\n{h}\n\n")).unwrap_or_default(),
        history_section = history_section,
    );
    
    let path = self.config.shared.join("ERROR_FEEDBACK.md");
    tokio::fs::write(&path, &content).await?;
    
    info!(
        path = %path.display(),
        source,
        attempt,
        "Wrote ERROR_FEEDBACK.md for agent self-repair"
    );
    
    self.error_feedback_attempts += 1;
    Ok(())
}
```

### 3. Error History (`error_history.json`)

A persistent JSON file in the shared directory that accumulates error context across attempts, so FORGE can see what was tried before and avoid repeating the same mistakes.

#### 3a. Structure

```json
{
  "entries": [
    {
      "timestamp": "2026-04-24T07:08:37Z",
      "source": "setup",
      "error_type": "merge_origin_main_failed",
      "message": "Merging is not possible because you have unmerged files",
      "resolution_attempted": null,
      "resolved": false
    },
    {
      "timestamp": "2026-04-24T07:12:00Z",
      "source": "build_verification",
      "error_type": "compilation_error",
      "message": "src/App.tsx(12,5): error TS2304: Cannot find name 'counter'",
      "resolution_attempted": "added missing import",
      "resolved": true
    }
  ]
}
```

#### 3b. `append_error_history()` method

```rust
async fn append_error_history(&self, source: &str, message: &str) -> Result<()> {
    let path = self.config.shared.join("error_history.json");
    
    let mut history = if path.exists() {
        tokio::fs::read_to_string(&path)
            .await
            .ok()
            .and_then(|s| serde_json::from_str(&s).ok())
            .unwrap_or(ErrorHistory { entries: vec![] })
    } else {
        ErrorHistory { entries: vec![] }
    };
    
    history.entries.push(ErrorHistoryEntry {
        timestamp: chrono::Utc::now().to_rfc3339(),
        source: source.to_string(),
        error_type: classify_error(message),
        message: truncate_message(message, 2000),
        resolution_attempted: None,
        resolved: false,
    });
    
    tokio::fs::write(&path, serde_json::to_string_pretty(&history)?).await?;
    Ok(())
}
```

#### 3c. Mark resolved on success

When FORGE writes STATUS.json with PR_OPENED after a rework cycle (meaning it resolved the error), mark the latest unresolved entry:

```rust
async fn mark_error_resolved(&self) -> Result<()> {
    let path = self.config.shared.join("error_history.json");
    if !path.exists() { return Ok(()); }
    
    let mut history: ErrorHistory = tokio::fs::read_to_string(&path)
        .await?
        .parse()?;
    
    if let Some(last_unresolved) = history.entries.iter_mut().rev().find(|e| !e.resolved) {
        last_unresolved.resolved = true;
        last_unresolved.resolution_attempted = Some("resolved_by_forge".to_string());
    }
    
    tokio::fs::write(&path, serde_json::to_string_pretty(&history)?).await?;
    Ok(())
}
```

#### 3d. Include in prompts

In both `new_session_prompt()` and `rework_prompt()`, read error_history.json and include a summary:

```rust
let error_history = self.read_error_history_summary(shared);
// Include in prompt:
// "## Error History (previous attempts)\n{error_history}\n\n\
//  IMPORTANT: Review the error history above. Do NOT repeat the same failed approach."
```

#### 3e. `cleanup()` preserves error_history.json

Error history is NOT removed during cleanup — it persists across pair lifecycles for the same ticket. It's only removed when the ticket is completed or marked Exhausted.

### 4. Escalation Path to Nexus

When the pair can't self-repair after max attempts, it returns `PairOutcome::Blocked` with detailed error context. This flows through the existing path:

```
PairOutcome::Blocked { reason, blockers } 
  → ForgePairNode::exec_one() returns json!({ "outcome": "blocked", "reason": ..., "blockers": ... })
  → post_batch() sets WorkerStatus::Suspended { reason }
  → Action returned: ACTION_FAILED → routes to nexus
  → Nexus sees Suspended worker with detailed reason
```

**Enhancement**: Add a `blocker_type` field to the `Blocker` struct so nexus can categorize errors:

```rust
pub struct Blocker {
    pub blocker_type: String,  // "verification_failure", "setup_error", "git_error", "unresolvable_conflict"
    pub description: String,
    pub nexus_action: String,  // "retry_with_different_approach", "escalate_to_human", "reassign"
}
```

This gives nexus structured information to reason about what kind of intervention is needed.
