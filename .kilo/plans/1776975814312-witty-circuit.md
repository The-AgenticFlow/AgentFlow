# Implementation Plan: LORE Agent (Documenter)

**Issue**: [#3 - feat(lore): Implement Lore Agent (Documenter)](https://github.com/The-AgenticFlow/AgentFlow/issues/3)
**Milestone**: Phase 3: Multi-Agent Coordination

## Overview

Implement the LORE agent crate for automated documentation generation. LORE is responsible for preserving institutional memory through ADRs, changelogs, and documentation-as-code management.

## Agent Definition (from lore.agent.md)

```yaml
id: lore
role: documenter
cli: claude
active: true
permissions:
  allow: [Read, Write, Bash, DocCommit]
  deny: [EditAppCode, EditInfraCode, Slack]  # LORE only writes docs/
```

**Non-negotiables**:
- Write an ADR for every architectural decision recorded in SharedStore
- Update `CHANGELOG.md` for every successful deployment
- Read SharedStore sprint history for contextual retrospectives
- Maintain high documentation clarity and formatting standards

## Architecture

### Crate Structure

```
crates/agent-lore/
├── Cargo.toml
└── src/
    ├── lib.rs           # Public exports and LoreNode implementation
    ├── types.rs         # LoreConfig, LoreOutcome, DocumentationTask
    ├── adr.rs           # ADR generation logic
    ├── changelog.rs     # Changelog automation
    ├── retrospective.rs # Sprint retrospective generation
    └── docs.rs          # Documentation-as-code utilities
```

### Integration Point

LORE integrates into the flow after VESSEL successfully merges/deployes:
- VESSEL emits `ticket_merged` event → NEXUS routes to LORE
- LORE reads merged ticket details from SharedStore
- LORE generates documentation, writes ADRs, updates changelog

## Triggers & Events

**Trigger Mode**: Asynchronous (fire-and-forget). LORE runs in the background after deployment, not blocking the deployment completion notification.

### When LORE is invoked:

1. **After successful deployment** (`ACTION_DEPLOYED`):
   - Update `CHANGELOG.md` with merged PR details
   - Check for architectural decisions → generate ADR if needed
   - Update README if feature documentation is stale

2. **After architectural decision recorded**:
   - SharedStore event with decision details
   - LORE generates corresponding ADR

3. **On sprint completion** (on-demand via NEXUS command):
   - Generate retrospective document
   - Aggregate sprint metrics

4. **README sync** (when relevant):
   - Update README feature sections
   - Sync installation instructions if changed
