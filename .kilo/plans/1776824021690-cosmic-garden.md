# Fix: Sentinel Retry Death Spiral & CI Fix Feedback Loop

## Problem Summary

The forge-sentinel pair enters a death spiral when a SENTINEL evaluation fails:
1. `sentinel_retries` is never reset after failure/timeout → stays at MAX
2. Every subsequent spawn attempt immediately hits the guard, increments `reset_count`
3. Rapid loop (~150ms) burns through all 10 resets → `FuelExhausted`
4. The "forcing changes_requested" log is hollow — no artifact is written, so FORGE never learns what failed
5. Forge can't reproduce CI failures locally because CI_FIX.md only has heuristic-based commands, not actual workflow definitions

The log evidence shows exactly this: 10 rapid "exceeded max retries" messages in ~1.5 seconds, then fuel exhausted.

## Root Causes

### RC-1: `sentinel_retries` Never Reset After Failure (Critical)
**File**: `crates/pair-harness/src/pair.rs`

`sentinel_retries` is only reset to 0 at line 462 (SENTINEL success). It is NOT reset when:
- SENTINEL exits with error (lines 463-470)
- SENTINEL times out (lines 481-490)
- The "force changes_requested" guard fires (lines 992-998, 1026-1033, 1067-1073)

Once `sentinel_retries` reaches `MAX_SENTINEL_RETRIES` (2), it stays there forever. Every subsequent call to any `spawn_sentinel_for_*()` immediately hits the guard, calls `reset.increment_reset()`, and returns without actually spawning SENTINEL. This creates the tight loop that burns through the reset budget.

### RC-2: "Forcing changes_requested" Writes No Artifact (Critical)
**File**: `crates/pair-harness/src/pair.rs`, lines 992-998, 1026-1033, 1067-1073

The guard logs "forcing changes_requested" but:
- Does NOT write `segment-N-eval.md` with verdict `CHANGES_REQUESTED`
- Does NOT write `CONTRACT.md` with `status: ISSUES`
- Does NOT write `final-review.md` with `REJECTED`
- Does NOT emit any event to the watcher

Result: FORGE never gets feedback. The harness stays in limbo. No `FsEvent` triggers FORGE re-spawning with fix instructions. The only side effect is `reset.increment_reset()`.

### RC-3: Shared `sentinel_retries` Counter Across Modes (Medium)
**File**: `crates/pair-harness/src/pair.rs`, field at line 256

All three sentinel modes (plan review, segment eval, final review) share a single `sentinel_retries` counter. A plan review failure makes segment eval and final review immediately hit the guard.

### RC-4: Forge Cannot Reproduce CI Locally — No Workflows, No Merge (Medium)
**File**: `crates/pair-harness/src/pair.rs`, `crates/agent-vessel/src/node.rs`, `crates/github/src/rest.rs`

The CI fix flow has two fundamental gaps:

1. **Forge branch doesn't have `.github/workflows/` from main** — if workflows were added/changed on main after the branch was created, FORGE doesn't have them and can't run the real CI steps. The current approach uses `local_reproduce_commands()` which keyword-matches check names to guess commands (heuristic, brittle, often wrong).

2. **No merge of origin/main before CI fix** — the conflict resolution path already merges `origin/main` into the worktree (pair.rs:377-408) so FORGE can see conflicts. The CI fix path does NOT do this. So FORGE: (a) doesn't have the latest workflow definitions, (b) doesn't detect merge conflicts that may be contributing to CI failure, and (c) gets heuristic commands instead of real ones.

The fix: merge `origin/main` before CI fix (same as conflict path), drop the heuristic `local_reproduce_commands()` entirely, and instruct FORGE to read the actual workflow files and run the real steps.

### RC-5: No Backoff on Sentinel Retry (Low)
**File**: `crates/pair-harness/src/pair.rs`

When SENTINEL fails and the harness retries, there's no delay between attempts. In the death spiral, the loop runs at ~150ms intervals (the 50ms sleep at line 804 + event processing).

## Fix Plan

### Fix 1: Reset `sentinel_retries` on All Exit Paths
**Priority**: P0 | **File**: `crates/pair-harness/src/pair.rs`

Reset `sentinel_retries` to 0 in these locations:

a) **SENTINEL timeout** (line ~490): After killing the timed-out sentinel, reset `sentinel_retries = 0`. A timeout is a transient failure; the next spawn should start fresh.

b) **SENTINEL error exit** (line ~469): After logging the error exit, reset `sentinel_retries = 0`. The error was already counted by the increment at spawn time; keeping it elevated just prevents the next attempt.

c) **"Force changes_requested" guard** (lines 998, 1033, 1073): After incrementing `reset_count`, reset `sentinel_retries = 0`. The guard already acted; leaving the counter at MAX just causes the death spiral on the next call.

d) **FORGE respawn** (all `spawn_forge_resume()` call sites): When FORGE is respawned with a fresh context, reset `sentinel_retries = 0`. The new FORGE session is a clean slate; sentinel evaluations should start fresh.

**Rationale**: The sentinel_retries counter should track "consecutive failures for THIS evaluation attempt." Once a context reset or forge respawn occurs, the evaluation context has changed, so the counter should reset.

### Fix 2: Write Synthetic `changes_requested` Artifacts
**Priority**: P0 | **File**: `crates/pair-harness/src/pair.rs`

When the max-retries guard fires, write a synthetic eval file so the lifecycle proceeds normally:

a) **Segment eval guard** (line 1026-1033): Write `shared/segment-{n}-eval.md` with:
   ```markdown
   # Segment {n} Evaluation

   ## Verdict: CHANGES_REQUESTED

   ## Specific feedback
   SENTINEL evaluation failed after {MAX_SENTINEL_RETRIES} attempts.
   FORGE must re-verify this segment's implementation locally before resubmitting.
   Run all tests and lint checks for this segment's code before updating WORKLOG.md.
   ```

b) **Plan review guard** (line 992-998): Write `shared/CONTRACT.md` with `status: ISSUES` and a similar synthetic feedback message.

c) **Final review guard** (line 1067-1073): Write `shared/final-review.md` with verdict `REJECTED` and synthetic feedback.

This ensures:
- The watcher detects a legitimate file event → FORGE gets feedback
- FORGE reads the eval file and knows it needs to fix issues
- The lifecycle doesn't stall in limbo
- The `FsEvent` handlers at lines 557-566 (SegmentEvalWritten) and 568-579 (FinalReviewWritten) fire normally

### Fix 3: Per-Mode Sentinel Retry Counters
**Priority**: P1 | **File**: `crates/pair-harness/src/pair.rs`

Replace `sentinel_retries: u32` (line 256) with a struct:

```rust
struct SentinelRetryState {
    plan_review_retries: u32,
    segment_eval_retries: std::collections::HashMap<u32, u32>,  // segment -> retries
    final_review_retries: u32,
}
```

Each `spawn_sentinel_for_*()` checks and increments only its own counter. Failures in plan review don't affect segment eval or final review, and different segments have independent retry budgets.

**Note**: If this is too invasive for a first pass, an acceptable alternative is to reset `sentinel_retries = 0` when transitioning between evaluation phases (i.e., when a new segment starts eval, when plan review completes and implementation begins, etc.). This is simpler but less robust.

### Fix 4: Merge `origin/main` Before CI Fix — Drop Heuristics Entirely
**Priority**: P1 | **Files**: `crates/pair-harness/src/pair.rs`, `crates/agent-vessel/src/node.rs`, `crates/github/src/rest.rs`

The current approach of heuristically generating local reproduce commands (`local_reproduce_commands()`) is fundamentally flawed — it keyword-matches check names and guesses commands. Instead, the forge pair should pull and merge `origin/main` into their branch, which:

1. **Brings the actual `.github/workflows/` files locally** — FORGE can read the real YAML, find the failing job, and run its exact `steps[*].run` commands
2. **Surfaces merge conflicts simultaneously** — if main changed files that conflict with the branch, FORGE sees the markers and can resolve them in the same pass
3. **Gets dependency/infra changes from main** — if CI was updated on main (new linter version, new dependencies), FORGE now has those changes

The conflict resolution path already does exactly this at `pair.rs:377-408` via `self.worktree.merge_origin_main()`. The CI fix path just needs the same treatment.

#### Changes:

a) **pair.rs — Merge origin/main when CI_FIX.md exists** (after line 408, add analogous block):
   ```rust
   // 1c. If CI fix rework: merge origin/main so FORGE has workflow files + detects conflicts
   if ci_fix_path.exists() {
       match self.worktree.merge_origin_main(&self.config.worktree) {
           Ok(MergeMainResult::Clean) => {
               info!(pair = %self.config.pair_id,
                   "origin/main merged cleanly before CI fix — FORGE has latest workflows");
               if let Err(e) = self.worktree.force_push_branch(&self.config.worktree) {
                   warn!(pair = %self.config.pair_id, error = %e,
                       "Failed to force-push after clean merge — GitHub PR may be stale");
               }
           }
           Ok(MergeMainResult::Conflict { conflicted_files }) => {
               info!(pair = %self.config.pair_id, files = conflicted_files.len(),
                   "Merge conflicts surfaced during CI fix prep — FORGE will resolve both conflicts and CI failures");
           }
           Err(e) => {
               warn!(pair = %self.config.pair_id, error = %e,
                   "Failed to merge origin/main before CI fix — FORGE may not have latest workflows");
           }
       }
   }
   ```

b) **pair.rs — Rewrite the CI_FIX.md task prompt** (lines 867-921): Remove the heuristic `local_commands` section. Instead, instruct FORGE to:
   - Look at `.github/workflows/` to find the actual CI workflow files
   - Identify the failing job by name (still provided by vessel from GitHub API)
   - Run the exact `steps[*].run` commands from the workflow YAML
   - Resolve any merge conflict markers found in the worktree
   - Fix all errors, commit, and push

   New prompt structure:
   ```rust
   format!(
       "Fix CI failures for ticket {}.\n\n\
        Branch: {}\n\n\
        The CI pipeline failed:\n{}\n\n\
        ## How to fix:\n\
        1. Read .github/workflows/ to find the failing workflow(s).\n\
        2. Identify the failing job by matching the check name above to the job name in the YAML.\n\
        3. Run the job's steps locally — use the exact `run:` commands from the workflow.\n\
        4. If merge conflict markers are present in any files, resolve them first.\n\
        5. Fix all errors found.\n\
        6. Run ALL failing checks again to confirm everything passes.\n\
        7. git add -A && git commit -m \"fix CI failures\" && git push\n\
        8. Write STATUS.json with status PR_OPENED.\n\n\
        CRITICAL RULES:\n\
        1. Run ALL failing checks locally, fix ALL errors, then push ONCE.\n\
        2. Resolve any <<<<<<< conflict markers BEFORE running checks.\n\
        3. You MUST update {}/WORKLOG.md as you work — the watchdog will kill your process if WORKLOG.md is not updated within 20 minutes.\n\n\
        If a PR already exists for this branch, do NOT create a new one — just push and update STATUS.json.\n\n\
        VALID STATUS.json status values: PR_OPENED, COMPLETE, BLOCKED, FUEL_EXHAUSTED, PENDING_REVIEW. \
        Do NOT use any other value — an invalid status will be treated as BLOCKED and your fix will be wasted.",
       ticket.id,
       WorktreeManager::branch_name(&self.config.pair_id, &ticket.id),
       failure_summary,
       self.config.shared.display(),
   )
   ```

c) **vessel node.rs — Simplify `write_ci_fix_md()`** (lines 1245-1374):
   - Remove the `local_commands` section derived from `failure_detail.local_reproduce_commands()`
   - Replace the "How to Fix — Run Checks Locally" section with instructions to look at `.github/workflows/` locally
   - Keep the `## Failed Checks` section (the check names are still valuable for FORGE to match against workflow jobs)
   - Keep the `## Job Log Output` section (the actual error output is still valuable)
   - The harness-level prompt in pair.rs already provides the detailed step-by-step instructions, so vessel's CI_FIX.md just needs the raw CI data

d) **github/rest.rs — Remove `local_reproduce_commands()`** (lines 736-771):
   - Delete the `local_reproduce_commands()` method from `CiFailureDetail`
   - Delete the `Display` impl's reliance on it if any
   - The method is now dead code — the harness-level prompt handles workflow discovery

### Fix 5: Add Sentinel Failure Diagnostics to HANDOFF.md
**Priority**: P2 | **File**: `crates/pair-harness/src/pair.rs`, `crates/pair-harness/src/reset.rs`

When sentinel fails (timeout or error exit), capture the failure context and include it in the next handoff:

a) New field on `ForgeSentinelPair`: `last_sentinel_failure: Option<SentinelFailureInfo>`
   ```rust
   struct SentinelFailureInfo {
       mode: SentinelMode,
       reason: String,  // "timeout after 120s" or "exit code 1"
       stderr_excerpt: Option<String>,  // last 500 chars of stderr
       timestamp: Instant,
   }
   ```

b) When writing handoff (in `synthesize_handoff()` or when FORGE writes HANDOFF.md), append a section:
   ```markdown
   ## Last Sentinel Failure
   
   Mode: SegmentEval(2)
   Reason: timeout after 120s
   Stderr excerpt: ...
   ```

This gives the next FORGE session concrete information about why SENTINEL failed, enabling it to adjust its approach.

### Fix 6: Add Minimum Retry Interval
**Priority**: P2 | **File**: `crates/pair-harness/src/pair.rs`

Track `last_sentinel_spawn_time: Instant` and enforce a minimum interval (e.g., 30 seconds) between consecutive sentinel spawn attempts. This prevents the rapid-fire loop even if the retry counter logic has other bugs.

```rust
const MIN_SENTINEL_RETRY_INTERVAL_SECS: u64 = 30;
// In each spawn_sentinel_for_*():
if let Some(last) = self.last_sentinel_spawn_time {
    let elapsed = last.elapsed().as_secs();
    if elapsed < MIN_SENTINEL_RETRY_INTERVAL_SECS {
        debug!("Sentinel retry too soon ({}s < {}s) — deferring", elapsed, MIN_SENTINEL_RETRY_INTERVAL_SECS);
        return Ok(());
    }
}
```

## Implementation Order

1. **Fix 1** (sentinel_retries reset) — stops the death spiral immediately
2. **Fix 2** (synthetic artifacts) — ensures the lifecycle can proceed after a retry exhaustion
3. **Fix 4** (merge origin/main + drop heuristics) — gives FORGE real workflows and surfaces conflicts
4. **Fix 5** (failure diagnostics in handoff) — gives FORGE context about what went wrong
5. **Fix 3** (per-mode counters) — prevents cross-mode retry contamination
6. **Fix 6** (retry interval) — safety net against future rapid-fire loops

Fixes 1+2 together resolve the immediate loop. Fix 4 addresses the user's observation that forge can't replicate CI failures locally — by merging origin/main, FORGE gets the actual workflow files and can run the real CI steps. Fixes 3+6 are hardening.
